Business Context

Ozi is 4–5 dark stores in.
The question isn't what to build — it's what to test first.

  • At this stage, the riskiest move is shipping a big feature nobody repeats
  • The goal isn't a product roadmap — it's signal: is repeat behavior emerging at all?
  • Limited data, limited runway, limited margin for wrong bets
  • Dark-store model means unit economics live or die on reorder frequency
  • Focus energy on reducing uncertainty before committing engineering capacity

ICP + User Pain

Two types of parents.
One shared frustration.

Emergency Parent
Ran out of diapers at 10pm. Chose Ozi for speed. Will choose whoever gets there first next time — unless Ozi catches them before the emergency.
Routine Replenishment
Buys monthly. Forgets to reorder until almost out. Low brand loyalty, high convenience loyalty. Currently forming habits — Ozi can capture this or lose it.
First-Time Parent
Unsure of sizes, brands, quantities. Needs guidance and trust, not a catalogue. Values feeling guided, not browsing.
Shared pain: No signal from Ozi that it's time to reorder. The app feels like a marketplace, not a replenishment partner — and every time they open Blinkit out of habit, Ozi loses the order.
AI Product OS
Hypothesis

One hypothesis. One test.
Three weeks.

"If Ozi triggers a reorder reminder 18–20 days after a diaper delivery and routes the user directly to a pre-filled repeat order, repeat purchase rate within 21 days will improve — because it removes the memory burden and the 'I'll-do-it-later' escape hatch."
  • Why testable now: small cohort, binary outcome, zero ML or personalization required
  • What it is NOT: a subscription product, a loyalty program, or a personalization engine
  • Decision in 3 weeks: did the test group reorder more than control? Yes → build. No → diagnose.

Experiment Design

MVP: Reorder Reminder + One-Tap Repeat

Cohort: first-time diaper buyers in last 60 days who have not reordered · 50/50 test vs. control

Trigger Mechanism
Option A — Event-based (if tooling exists)
  • Hook into order_delivered webhook
  • Schedule push at delivered_at + 18–20 days via OneSignal / CleverTap / Braze
  • No new cron infrastructure needed
Option B — Cron-based (no tooling needed)
  • Daily cron queries orders delivered 18–20 days ago
  • Fan-out per user with Promise.allSettled
  • Hard cap 500/run · Full observability via cron_runs table
Repeat Order Path
Option A — Pre-filled cart deep link
  • Notification encodes order_id
  • Ozi app resolves → pre-populated cart
  • One "Place Order" tap → done
Option B — Lightweight repeat screen
  • Deep link to /reorder/:orderId
  • Single SKU card, quantity pre-set to last order
  • "Add to Cart" → standard Ozi checkout

Both paths: user never browses. One tap, pre-filled, done.

Metrics + Instrumentation

Zero-to-one metrics: what to track when you have no history

Metric Type Target
21-day repeat purchase rate — test vs. control North Star +10pp lift
Notification open rate Supporting >25%
Repeat screen → order conversion Supporting >40%
Notification opt-out rate Guardrail Alert if >5%
reminder_triggered → reminder_delivered → reminder_opened → cart_prefilled → checkout_started → order_placed
  • These 7 events cover the full funnel and isolate exactly where drop-off happens
  • If CTR >10% and completion >50% — the mechanic works, scale it

RCA Framework

If it fails, here's exactly how we find out why

Each failure mode maps to a specific next action. Nothing goes into a dead backlog.

Signal
Repeat rate ~0 in both groups
Root cause: wrong segment. First-time buyers may not be repeat buyers yet.
Next: test with subscription-intent parents or higher-frequency SKUs.
Signal
Reminder CTR <5%
Root cause: wrong timing or wrong copy. Day 18 too early or too late.
Next: A/B test Day 14 vs. Day 21.
Signal
CTR decent, cart abandonment high
Root cause: message created an expectation the UX didn't deliver.
Next: rewrite copy, fix cart pre-fill accuracy.
Signal
Pre-fill shows wrong size / product
Root cause: UX friction — deep-link payload inaccurate.
Next: fix payload, add size confirmation step.
Signal
repeat_order_failed spikes
Root cause: operational problem, not a PM problem.
Next: escalate to ops and dark-store leads.
Loop
Every outcome → next hypothesis
RCA output → tagged backlog item → sharper experiment → new test cycle. The experiment never truly fails.
