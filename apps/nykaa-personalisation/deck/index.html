Architecture

System design — built for debuggability

Homepage Search Page ForYouShelf SearchResults | | v v GET /api/shelf --------+-----> GET /api/rerank | | | PersonalisationSvc | RerankEngine | | | CohortSvc (SHA-256) | // Same scoring | | v v Neon DB (3 tables) experiment_cohorts | user_affinity_profiles | session_events User Click ----> POST /api/ingest ----> session_events (24h TTL) Nightly Cron --> POST /api/rebuild --> user_affinity_profiles
Choice Rationale
Neon DB (not Supabase) Direct PostgreSQL, no ORM overhead, serverless-native
PostHog (not Mixpanel) Open-source, A/B cohort analysis built-in
Rule-based scoring (not ML) Debuggable, explainable, no training cycles
Fire-and-forget telemetry Observability never degrades user experience

Scoring Engine

How products get ranked — the math

Final Score = 0.6 x Affinity Score + 0.4 x Intent Score Affinity Score = (brand_match + category_match) / 2 brand_match: 1 if product brand in user's top 5 brands, else 0 category_match: 1 if product category in user's top 5 categories, else 0 Source: Order history, rebuilt nightly by AffinityBuilder Intent Score = (brand_intent + category_intent) / 2 brand_intent: 1 if product brand matches any of last 3 clicks, else 0 category_intent: 1 if product category matches any of last 3 clicks, else 0 Source: session_events table (24h TTL)
Example: User "Priya" (Fashion)
Top brands: Zara, H&M, Mango. Last 3 clicks: Nike sneaker, Zara top, H&M blazer.

H&M Blazer: 0.6 x 0.5 + 0.4 x 1.0 = 0.70 (ranked high)
Lakme Lipstick: 0.6 x 0.0 + 0.4 x 0.0 = 0.00 (ranked low)
Why 60/40? Historical affinity represents durable preferences. Session intent captures ephemeral style goals (e.g., shopping for a wedding guest dress even though usual style is casual). 60/40 anchors on stable signal while remaining responsive to in-session behaviour.

Experiment Design

Deterministic A/B testing — SHA-256 cohort split

cohort = SHA-256(userId + serverOnlySalt) % 2 0 = Control (editorial default) 1 = Test (personalised ranking)
  • Reproducible: Same user always gets same cohort (no cookie dependency)
  • Stable: Assignment persisted in DB (write-once, first-assignment-wins)
  • Secure: Salt is server-only (never exposed to browser bundle)
  • Masked: Control group API response says "default" not "control" to prevent experiment contamination

Metrics

North Star: ATC Rate Lift

ATC is unmeasurable in current MVP (no PDP/cart flow). Shelf CTR is the measurable proxy.

Metric Target Why It Matters
Shelf CTR Lift North Star Proxy +15% Are recommendations relevant enough to click?
Search Re-rank CTR Supporting +10% Does re-ranking improve search quality?
Shelf Load Latency (P95) Supporting ≤ 200ms Personalisation must not degrade UX
Cold-Start Suppression Rate Guardrail Track How many users get fallback due to insufficient data?

Funnel

Homepage Visit → Shelf Impression → Shelf Click → PDP View → ATC
  • P95 > 500ms → investigate (AbortController fires, users see fallback)
  • shelf_load_failed > 2% of impressions → investigate
  • Cohort split outside 48/52 → investigate assignment algorithm
